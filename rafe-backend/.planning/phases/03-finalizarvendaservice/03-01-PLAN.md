# PLAN 03-01: Domain, Ports e Fakes para FinalizarVendaService

Este plano visa preparar o terreno para o `FinalizarVendaService`. A orquestração deste serviço exige garantir a atomicidade das operações e requer que o `DocumentoFiscal` possua um identificador, visto que o módulo de pagamentos precisa associar o pagamento ao ID do documento.

## Tarefas a Executar

### 1. Criar TransacaoPort e FakeTransacao
- Criar a interface `App\Modules\POS\Domain\Ports\Saida\TransacaoPort` com a assinatura `executar(callable $operacao): mixed`.
- Criar `App\Modules\POS\Adapters\Saida\Fakes\FakeTransacao` que implemente este Port. A implementação do Fake deve simplesmente invocar `$operacao()` e repassar as excepções geradas (simulando que a transacção não fez commit em caso de erro).

### 2. Actualizar DocumentoFiscal e respectivo Port
- No `App\Modules\POS\Domain\DocumentoFiscal`, adicionar a propriedade `?int $idDocumento = null` com os métodos `idDocumento(): ?int` e `definirIdDocumento(int $id): void`.
- Em `App\Modules\POS\Domain\Ports\Saida\DocumentoFiscalRepositorioPort`, alterar o método `salvar(DocumentoFiscal $documento)` para que retorne `int` em vez de `void`. Adicionar também `proximoNumeroDocumento(): int`.

### 3. Implementar FakeDocumentoFiscalRepositorio
- Actualizar `FakeDocumentoFiscalRepositorio` para que o método `salvar` persista os dados na `$this->tabela_venda`, preenchendo as colunas necessárias à semelhança do que fizemos no `FakePedidoRepositorio`.
- Implementar o método `proximoNumeroDocumento()` a devolver um valor estático para os testes (ex: `1`).

## Regras
- Validar a sintaxe de todos os ficheiros modificados e novos com `php -l`.
- Garantir que não são introduzidas dependências do Eloquent nem do `DB::transaction()` do Laravel.
